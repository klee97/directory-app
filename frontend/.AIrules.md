# AI Coding Rules

## Project Overview
HMUA Directory — a Next.js 16 (App Router) marketplace for Asian wedding makeup artists. Multi-tenant: separate public customer routes and vendor-only routes. Backend is Supabase (PostgreSQL + auth + storage), with Contentful for blog/CMS content.

## Tech Stack
- **Framework**: Next.js 16, React 19, TypeScript (strict)
- **Package manager**: pnpm
- **UI**: Material-UI (MUI) v6 + Emotion
- **Database/Auth**: Supabase (PostgreSQL, RLS, PostGIS)
- **CMS**: Contentful (GraphQL)
- **Image storage**: Cloudflare R2 (S3-compatible)
- **Testing**: Vitest + React Testing Library + MSW

## Coding Conventions

### TypeScript
- Strict mode is on — no implicit `any`
- Always use `@/` path aliases for imports from `src/` (e.g., `@/lib/supabase/server`)
- Auto-generated GraphQL types live in `src/lib/generated/` — do not edit manually

### Components
- PascalCase filenames for components (e.g., `BackButton.tsx`)
- camelCase for utilities and hooks (e.g., `passwordValidation.ts`)
- Add `"use client"` directive only when needed (interactivity, hooks, browser APIs)
- Default to Server Components for data fetching

### Supabase
- Use `createClient()` from `@/lib/supabase/server` in Server Components and API routes
- Use `createClient()` from `@/lib/supabase/client` only in Client Components
- Never bypass RLS — use the appropriate client (not the service role key) in user-facing code
- Service role client is only for trusted server-side operations (admin scripts, migrations)

### Feature Organization
- New features go in `src/features/[featureName]/`
- Each feature can have its own `components/`, `hooks/`, `types/`, `utils/` subdirectories
- Shared UI primitives go in `src/components/ui/`

### Styling
- Use MUI `sx` prop or `styled()` from `@mui/material/styles` — avoid plain CSS files
- Follow the existing MUI theme in `src/components/theme/`

### State Management
- Use React Context for global state (see `src/contexts/`)
- `useAuth()` hook for auth state — do not access Supabase auth directly in components
- `useContext(NotificationContext)` for toast notifications

## Auth Patterns
In general, middleware (`proxy.ts`) handles fast access control (hard gates before the page renders).
Server-side auth checks using `getCurrentUser()` from `@/lib/auth/getUser` provide additional security.
Client-side `AuthContext` should be generally used for UI personalization and condtional rendering

### Client Components
- Use `useAuth()` from `@/contexts/AuthContext` for auth state in two situations:
  - **Conditional UI on public pages** — e.g. showing a login button vs. user avatar in the header
  - **User object access in deeply nested components** — avoids prop drilling the session down the tree
- Use `isAdminRole(role)`, `isVendorRole(role)` from `@/lib/auth/userRole` for role checks

### Server Components / Server Actions / API Routes
- In most cases, use `getCurrentUser()` from `@/lib/auth/getUser`. This makes an under-the-hood `supabase.auth.getClaims()` that is cached per request.
- Use `supabase.auth.getUser()` only when you need guaranteed server-side session validity (token revocation)

### Sensitive mutations (password / email change)
- `getClaims()` to extract current email → `signInWithPassword` to re-authenticate → `updateUser`
- Do not use `getUser()` here — security comes from the re-auth step, not token verification

### Supabase Clients
Use this table to determine which Supabase client in `lib/supabase/clients` is the right fit:

| Client | File | Key | Use when |
|---|---|---|---|
| Browser | `browserClient.ts` | publishable | Client components, browser-side mutations (auth, DB) |
| Server | `serverClient.ts` | publishable | Server components, server actions, route handlers — reads session from cookies |
| Admin | `adminClient.ts` | service role | Privileged server-only ops that bypass RLS — never import in client code |
| Static | `staticClient.ts` | publishable | Build-time and cached contexts with no request/cookie context — `generateStaticParams`, ISR, cached fetch helpers |

### Middleware
- Autowired by Next.js using `src/proxy.ts`
- Handles session refresh on every request via `getClaims()`
- Protects routes in `PROTECTED_PATHS` — redirects unauthenticated users to login
- Redirects authenticated users away from login pages to their respective home

## Testing
Unit tests (Vitest + React Testing Library):
- Tests live next to the code they test: `Component.test.tsx` or `Component.spec.tsx`
- MSW (`tests/msw/`) mocks API calls — add handlers there for new endpoints
- Use React Testing Library (`@testing-library/react`) for component tests
- Test user credentials (local Supabase only): `test-user@example.com` / `Testpassword123!`

E2E tests (Playwright):
- Live in `/e2e` at the repo root (not inside `src/`)

## Environment & Feature Flags
- Env vars follow Next.js conventions: `NEXT_PUBLIC_*` for client-safe vars
- Feature flags: `NEXT_PUBLIC_FEATURE_FAVORITES_ENABLED`
- Vendors with IDs starting with `TEST-` are filtered out in production
- Never commit `.env*` files

## Images
- Multiple image sources: Contentful CDN, Supabase storage, Cloudflare R2
- Image processing via Sharp (server-side)
- Use existing image utility functions in `src/lib/images/` — do not write raw S3 calls

## Analytics
- Google Tag Manager, Vercel Analytics, Microsoft Clarity — all conditional on environment
- Analytics calls are in `src/utils/analytics/` — use those helpers, don't call GTM directly

## Do Not
- Do not edit files in `src/lib/generated/` (auto-generated from GraphQL codegen)
- Do not use the Supabase service role key in client-side code
- Do not add `console.log` statements in committed code
- Do not skip `eslint` or TypeScript checks (`--no-verify`, `// @ts-ignore`) without a comment explaining why
- Do not use `any` type — use `unknown` and narrow, or look up the correct Supabase/MUI types
